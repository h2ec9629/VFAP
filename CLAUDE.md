ローカルで作業が完了した後にワンドライブへコピーを置くこと

## OneDriveフォルダ下のファイル全般の編集ルール

app.py に限らず、このOneDriveフォルダ配下のファイル全般で
OneDrive同期による「ファイル途中切れ（truncation）」が発生し得る。
（例：日程ページ_試作.html 編集中、サンドボックス側が同期途中の
不完全なファイルを掴み node の構文チェックが誤って失敗した）

対応方針：
- 作業は outputs（ローカル）側で行い、完了後に OneDrive へ書き出す
- 書き込み後の検証は、shell（bashマウント）の読み取りが同期遅延で
  古い断片を返すことがあるため、file ツール（Read）側の内容を正とする
- 構文・整合性チェックで失敗したら、まず実害か同期遅延かを切り分ける
  （file ツールで末尾まで揃っているか確認 → 揃っていれば同期遅延）

## ★重要★ OneDrive → outputs へのコピー手順（毎回必須）

bash の cp コマンドは OneDrive 同期途中の不完全なファイルを掴む。
セッションをまたいで繰り返し同じ破損が起きているため、以下を厳守すること。

### ファイルをコピーする前に必ずUTF-8チェックを実行する

```python
python3 -c "
path = '/sessions/.../mnt/おシャンティ　DB作成/対象ファイル.html'
try:
    open(path, encoding='utf-8').read()
    print('OK')
except Exception as e:
    print(f'壊れてる: {e}')
"
```

- **OK** → cp でコピーしてよい
- **壊れてる** → cp を使わず Read ツールで読んで Write ツールで outputs に書き出す

### コピー方法の優先順位

1. **チェックOKの場合のみ** `cp`（bash）を使う
2. チェックNGまたは不明な場合は必ず **Read → Write** を使う
   （Read ツールはクラウドから直接取得するため完全なファイルが得られる）

## ★重要★ Edit ツール使用後の末尾チェック（必須）

Edit ツールで outputs のファイルを編集した後も、ファイルが途中で切れることがある
（OneDriveと無関係、Edit ツール自体の書き込みが途中で終わるケースあり）。

**Edit ツールで変更したら必ずすぐ bash で末尾を確認すること：**

```bash
python3 -c "
with open('対象ファイルのbashパス', encoding='utf-8') as f: c = f.read()
print('size:', len(c))
print(repr(c[-80:]))
"
```

- 末尾が `</html>` や `}` など正常な終端なら OK
- 途中で切れていたら Python の `append` モードで不足部分を補完する

## app.py 編集ルール

OneDrive同期・linterによるファイル切断が頻発するため、
Editツールを使わず Pythonスクリプト（src.replace）で一括編集すること。
手順：
1. バックアップ（outputs/app.py.bak）を読み込む
2. replace で全差分を適用
3. ast.parse で構文チェック
4. 問題なければ OneDrive側に書き込む

作業後は変更点を簡潔に述べる

## HTMLのレイアウト・サイズ調整時の注意（全ページ共通ルール）

★ kiroku.html に限らず、このプロジェクトで編集・新規作成する
　全てのHTMLページ（既存ページの修正、新規ページの制作いずれも）に適用すること。

レイアウトや行の高さ・パディングなどを変更しても見た目に反映されない場合、
「キャッシュのせい」と決めつける前に、まずグローバルな強制スタイルが
競合していないか疑うこと（同一ファイル内・他の共通CSSファイル問わず）。

新規ページを作る時も、最初からテーブル名・クラス名で絞ったセレクタを使い、
グローバルセレクタ（`td:has(> input)`等、要素名だけ・絞り込みなしのもの）の
乱用を避けること。

実例（kiroku.html）：
- 131〜134行目あたりに `td:has(> input) > input{...padding:8px 5px;min-height:32px;...}`
  `td:has(> textarea) > textarea{...}` というテーブル名で絞っていない
  グローバルルールがあり、しかも一部 `!important` 付き。
  特定テーブル（kr-fry, kr-work等）の行高さ・パディングを調整しても、
  この全体ルールに上書きされて変化しないことがあった。
- `.kr-row-special` のように `!important` で背景色等を強制しているクラスもある。

対応方針：
- サイズ調整がうまく反映されない時は、`grep`で
  `!important|min-height:[0-9]|height:[0-9]`等を検索し、
  対象要素に効いているグローバルルールを洗い出す
- 上書きする際は `.テーブル名 td:has(> input) > input{...}` のように
  クラス名で絞り、相手が `!important` を使っていれば自分も付ける
- `display:flex` を `<td>`/`<th>` 自体に直接当てない
  （テーブルの列・colspan計算が崩れる原因になる。
  flexにしたい場合はセル内側の `<div>` に付けること）
